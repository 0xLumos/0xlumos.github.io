How Claude Code Actually Works

The agentic loop, streaming tool execution, context management, error recovery, and the permission model. Everything that happens between you pressing Enter and getting a result.

The Big Picture

Claude Code is a TypeScript application that renders a terminal UI with React and Ink. When you type a message, it gets sent to the Claude API along with a system prompt and a list of tool definitions. Claude responds with text, tool calls, or both. Tool calls execute locally on your machine. The results get fed back into the conversation. The loop repeats until Claude has nothing left to do.

There is no remote execution server. Your files, shell, and credentials stay on your machine unless a tool explicitly reaches out (like WebFetch or an MCP server).

That is the simple version. Here is what actually happens.

The Agentic Loop

Every interaction, whether you are in the interactive REPL or running a headless claude -p command, follows the same cycle. The core loop lives in query.ts, wrapped by QueryEngine.ts which adds session tracking, cost accounting, and SDK integration.

Your message enters the system
You type something in the REPL or pass it via --print / stdin. The message gets appended to the conversation history. If you are resuming a session, the full history is loaded from disk first.
Context is assembled
Before hitting the API, the system builds a prompt from several layers: your git status (branch, recent commits, working tree state), all discovered CLAUDE.md memory files, the current date, and the full list of available tools. This context is memoized for the session and only rebuilt if explicitly invalidated.
Pre-flight checks run
Before the API call, the system runs up to four compaction strategies to keep the conversation within the context window: history snipping (remove old messages cheaply), microcompaction (truncate stale tool results), auto-compaction (Claude summarizes older messages via a forked subprocess), and context collapse (project a compressed view over the full history). If the context is still too large after all that, the request is blocked before it wastes an API call.
The API streams a response
The conversation is sent to the Claude API and the response streams back token by token. As content blocks arrive, they are rendered to your terminal in real time. When a tool_use block arrives, something important happens: the tool starts executing immediately, while the stream is still running. This is a key latency optimization. The system does not wait for the full response before acting on tool calls.
Tools execute (with permission checks)
Each tool call passes through a six-step permission cascade before running. If approved, the tool executes and its output becomes a tool_result block in the conversation. If denied, Claude gets an error message explaining why. Multiple concurrent-safe tools (like file reads and grep searches) can run in parallel. Non-concurrent tools (like Bash) get an exclusive lock.
The loop decides what to do next
Once all tool results are collected, the system checks 11 possible exit conditions: normal completion, max turns reached, budget exhausted, context overflow, user abort, and others. If none are met and Claude's response contained tool calls, the loop iterates with the updated message history. If Claude responded with only text and no tool calls, the conversation turn is done.

The loop is a while(true) async generator that yields streaming events. It is not recursive. State is carried forward via a mutable State object that gets reconstructed at each iteration boundary. The generator pattern lets the UI consume events incrementally without buffering the entire turn in memory.

Streaming Tool Execution

This is probably the most interesting piece of the architecture. Most agent frameworks wait for the model to finish its entire response, then execute all the tool calls, then send the results back. Claude Code does not do that.

As soon as a tool_use block arrives in the stream, a StreamingToolExecutor picks it up and starts executing it. While the model is still generating the rest of its response (maybe more tool calls, maybe text), the first tool is already running. This overlaps model I/O with tool execution and meaningfully reduces end-to-end latency on multi-tool turns.

How Concurrency Works

Not all tools can run at the same time. Each tool declares whether it is concurrent-safe by implementing an isConcurrencySafe() method. The executor enforces a simple rule:

Scenario What happens
All queued tools are concurrent-safe They run in parallel. Read, Grep, Glob, and WebSearch are all concurrent-safe.
A non-concurrent tool is queued It waits until all running tools finish, then runs alone. Bash is the main example since shell commands can have side effects that conflict.
User aborts mid-execution An abort signal fires. Running tools are cancelled. Tools that were queued but never started get synthetic tool_result blocks saying "Interrupted by user" so the conversation stays consistent.

The Fallback Problem

Sometimes the system needs to switch models mid-stream (for example, from Opus to Sonnet if streaming overload is detected). When that happens, any tool executions from the old stream become orphans. The executor handles this by discarding the old executor, tombstoning any partial assistant messages so the UI removes them, stripping thinking blocks that might be incompatible with the new model, and spinning up a fresh executor for the retry.

Context Management

The context window is finite. Long conversations fill it up. Claude Code has a layered system to deal with this, and it is more sophisticated than most people realize.

Layer 1: History Snip
The cheapest strategy. Removes the oldest messages entirely from the API payload. The raw transcript on disk is never touched. This fires first and frees the most space with the least compute.
↓
Layer 2: Microcompaction
Targets stale tool results specifically. If a file was read 30 messages ago and has not been referenced since, the full content gets replaced with a short stub. This is done per-message and defers cache invalidation until after the API response to preserve prompt cache efficiency.
↓
Layer 3: Auto-Compaction
The heavy strategy. Spawns a forked Claude process that reads the older portion of the conversation and produces a natural-language summary. The summary replaces the originals in the API payload. The full transcript is preserved on disk. This costs an extra API call but can recover thousands of tokens.
↓
Layer 4: Context Collapse
A projection-based approach. Instead of rewriting history, it maintains a compressed "view" over the full message list and swaps it in at read time. Summaries live in a separate store. This is the most experimental strategy and is feature-gated.

These layers are not mutually exclusive. In a single pre-flight pass, the system might snip old messages, microcompact stale tool results, and then decide auto-compaction is still needed. Each layer has its own threshold and circuit breaker to prevent runaway compaction loops.

What happens when all layers fail? If the context is still too large after every compaction strategy has run, the system blocks the API call entirely and surfaces an error. It does not silently drop messages or make a doomed request. There is also a reactive compaction path that kicks in if the API itself returns a 413 (prompt too long), attempting one last recovery before giving up.

The Tool System

Claude Code ships with 40+ built-in tools and supports unlimited external tools via MCP. Every tool implements the same Tool<Input, Output, Progress> interface, which keeps the system uniform whether you are looking at a simple file read or a complex MCP integration.

What a Tool Looks Like Internally

Each tool has a Zod input schema for parameter validation, a call() method that does the actual work, and five rendering methods that control how it looks in the terminal at different points in its lifecycle: when called, when completed, while running, when the user rejects it, and when it errors.

Tools also declare metadata that affects how the system treats them:

Property What it controls
isConcurrencySafe() Whether this tool can run in parallel with others. Defaults to false (fail-closed).
isReadOnly() Whether the tool modifies anything. Read-only tools get lighter permission checks.
maxResultSizeChars When output exceeds this, it gets saved to a temp file and the model receives a preview with the file path. Prevents one large tool result from flooding the context window.
interruptBehavior() What happens if the user presses Ctrl+C. 'cancel' kills the tool; 'block' keeps it running.
checkPermissions() Tool-specific permission logic that runs before the general permission cascade.

All tools use a buildTool() factory that provides safe defaults, so you cannot accidentally ship a tool that forgets to declare its concurrency safety (it defaults to unsafe).

Result Size Management

This is a detail that matters more than it sounds. If a tool returns 500KB of text, shoving all of that into the conversation would eat the context window. The system handles this with a three-tier approach:

Tier 1: If the output is under maxResultSizeChars, send it all to the model as-is.

Tier 2: If it is over the limit, save the full output to a temp file on disk and send the model a preview (first ~1KB) plus the file path.

Tier 3: An aggregate budget tracks total tool result bytes across all tools in a turn. Even if each tool individually stays under its limit, the combined output can still be capped to prevent token explosion from many medium-sized results.

The Permission Model

This is where Claude Code is most opinionated. Every single tool invocation goes through a six-step permission check before anything executes. The system defaults to denying, not allowing.

Input validation
The tool's own validateInput() runs first. This catches things like invalid file paths, blocked device paths (/dev/zero, /dev/random), or nonsensical arguments. Rejected before any permission prompt is shown.
Deny rules
Checks against configured deny rules from all scopes (user settings, project settings, enterprise policy, session). A blanket deny like "Bash" blocks all shell commands. A specific deny like "Bash(rm -rf *)" blocks just that pattern.
Tool-specific permission logic
Some tools have custom checks. FileEdit runs write-permission checks against the file path. Bash parses the command into an AST and evaluates each subcommand individually against permission rules. This is proper shell parsing, not regex matching.
Rule-based evaluation
Allow and deny rules from all configured scopes (session > project > user > policy) are evaluated in priority order. The first matching rule wins.
Mode-based adjustment
The active permission mode (default, auto, bypass, plan) can override the decision. In bypass mode, everything is auto-approved. In default mode, unrecognized operations prompt the user.
Optional classifier
In auto mode, a background classifier can speculatively pre-approve Bash commands. It runs in parallel with the permission prompt, and whichever resolves first wins. If the classifier approves before you respond, the tool runs immediately.

Anti-fatigue protection. If you deny a tool 5 times in a row, the system stops asking and auto-rejects subsequent attempts for that tool. This prevents the model from getting stuck in a loop where it keeps retrying a denied action.

Three Handlers for Three Contexts

The permission system is not monolithic. There are three separate permission handlers depending on where the code is running:

Interactive
The main REPL. Shows a confirmation dialog in the terminal. Supports keyboard navigation to approve, deny, or grant session-wide permissions.
Coordinator
Used by the swarm leader. Runs automated checks before showing UI dialogs. Limits prompt frequency to avoid flooding the user with permission requests from multiple agents.
Swarm Worker
Background agents that cannot show UI. Auto-denies all interactive permission requests. Only hook-based approvals work in this context. If a worker needs something it does not have permission for, it fails and reports back to the leader.

Error Recovery

The agentic loop has to handle a lot of failure modes gracefully. The system uses a multi-layer recovery approach with circuit breakers at each level to prevent infinite retry loops.

When the Context Is Too Large (413)

If the API returns a 413 "prompt too long" error, the response is withheld (stored internally but not shown to the user or yielded to the UI). The system then attempts recovery:

First attempt: Context collapse drain. If the context collapse system is available, try projecting a more compressed view and retrying. This is the cheapest recovery.

Second attempt: Reactive compaction. Run a full summarization pass on older messages, then retry. This costs an API call but can recover a lot of space.

If both fail: Surface the error to the user. The system does not silently drop messages or try progressively more aggressive truncation. It stops and tells you what happened.

When the Output Gets Cut Off

Sometimes Claude's response hits the maximum output token limit mid-sentence. The system handles this in two stages:

Escalation: If the response was capped at the default 8K output limit and the feature is enabled, automatically retry with a 64K limit. This fires once per turn.

Multi-turn continuation: If the response is still too long at 64K, inject a synthetic user message asking Claude to continue where it left off. This can repeat up to a configurable limit, after which the system gives up and shows what it has.

When Models Fail Mid-Stream

If the primary model encounters a streaming error (overload, timeout, etc.) and a fallback model is configured, the system switches models mid-conversation. This involves tombstoning partial messages from the failed stream, stripping model-specific thinking blocks that might be incompatible, creating a fresh streaming executor, and notifying the user about the switch. The conversation continues seamlessly on the fallback model.

MCP Integration

The Model Context Protocol lets you connect external tools and data sources. Claude Code's MCP client supports multiple transport types (stdio, SSE, WebSocket, HTTP) and handles authentication, including OAuth with automatic token refresh and step-up auth detection.

MCP tools are wrapped in the same Tool interface as built-in tools. They go through the same permission cascade, the same result size management, and the same concurrency model. From the agentic loop's perspective, there is no difference between a built-in Read tool and an MCP tool from a remote server.

One implementation detail worth knowing: when assembling the tool list for the API call, built-in tools are sorted separately from MCP tools before concatenation. This preserves prompt cache stability. If MCP tools were interleaved with built-in tools, adding or removing a single MCP connection would invalidate the entire prompt cache.

Output sanitization. All MCP tool outputs are passed through a Unicode sanitizer that strips control characters and zero-width sequences. This prevents a malicious MCP server from injecting terminal escape codes that could manipulate your terminal display.

Sessions and Persistence

Every conversation is saved as a JSON transcript file in ~/.claude/. Each session has a unique ID. You can resume with --resume <id> or just --continue to pick up the most recent one.

When you resume: the full message history loads from disk. Memory files (CLAUDE.md) are re-discovered and may have changed since the original session. The permission mode resets to your configured default. Cost tracking state is restored so the running total stays accurate across sessions.

Compaction only affects what gets sent to the API. The raw transcript on disk always contains the full, uncompacted conversation. You never lose history to compaction.

Multi-Agent Coordination

Claude Code can spawn sub-agents via the Agent tool. Each sub-agent runs its own isolated agentic loop with a separate conversation, and optionally a restricted tool set. The leader agent orchestrates; workers execute.

Workers run with the swarm worker permission handler, which auto-denies interactive prompts. They can only do things that are pre-approved by rules or hooks. If a worker hits a permission wall, it fails gracefully and reports back to the leader, which can decide how to proceed.

In coordinator mode, workers get a subset of tools. They cannot spawn their own sub-agents (no recursive spawning) and do not have access to internal coordination tools. Each worker can optionally run in an isolated git worktree to avoid file conflicts with the leader or other workers.

Cost Tracking

Token usage is tracked per-model across the entire session. The cost tracker records input tokens, output tokens, prompt cache read/creation tokens, tool invocation counts, web search requests, and file change metrics (lines added/removed). All of this is visible via the /cost command.

A USD budget can be set per-session. If the running cost exceeds the budget, the agentic loop stops and reports the overage. Budget tracking survives session resume since the cost state is serialized to disk alongside the transcript.

The Entry Point

main.tsx is where everything starts. It parses CLI arguments, determines the execution mode (interactive, headless, resume, assistant, SSH remote, teleport, or direct connect), assembles the system prompt, initializes MCP connections and plugins, presents the trust dialog, and mounts the Ink application.

The system prompt is built in layers: a base prompt, optional teammate instructions, browser integration hints (for Chrome mode), custom agent prompts (for tmux-spawned agents), proactive mode addendum, and assistant mode addendum. Each layer is appended via string concatenation in sequence.

Feature flags. Many subsystems are gated behind compile-time feature flags using bun:bundle. When a feature is disabled, its code is eliminated at build time, not just branched around at runtime. This keeps the shipped bundle lean for each platform target (CLI, VS Code extension, desktop app, SDK).
